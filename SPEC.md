# Nokori SPEC

## アプリ概要

### アプリ名
Nokori

### コンセプト
つい使ってしまう支出をすばやく記録し、今月あといくら使えるかをシンプルに見える化する予算管理アプリ。  
日常の入力負荷はできるだけ低く保ちつつ、残額確認・履歴確認・振り返りを自然に行える体験を目指す。

### 提供価値
- 支出をタグ単位ですばやく記録できる
- 今月の予算 / 支出合計 / 残り金額をすぐ確認できる
- 月ごとの履歴を見返して使い方を振り返れる
- Premium で全期間履歴・残額ペース予測・月比較が使える

### 特徴
- ローカル完結のシンプルな家計管理
- グローバル月ナビゲーションによる一貫した月切り替え
- タグ起点の素早い支出登録
- 無料でも日常利用は十分可能、深い振り返りは Premium で拡張
- MVVM 前提で段階実装しやすい構成

---

## F-01 データモデルとローカル永続化基盤

### 目的（Goal）

Nokori の全機能が依存できる、ローカル完結のデータモデルと永続化基盤を定義する。月予算、タグ、支出、選択中の月を端末内に保存し、以降の F-XX ではこの基盤を利用して UI と操作を実装できる状態にする。

---

### ユーザーストーリー（User Story）

ユーザーとして、
アプリを閉じても予算・タグ・支出・今見ていた月が保持されていて、
次に開いたときも前回の続きから使いたい。

---

### 対象範囲（Scope）
- 端末内のみで完結する永続化基盤を用意する。クラウド同期は行わない。
- 以下の永続化対象を定義する。
  - 月予算
  - タグ
  - 支出
  - 選択中の月
  - タグの並び順
- 支出モデルを定義する。最低限以下の項目を持つ。
  - 一意な識別子
  - タグ識別子
  - 金額
  - 日付
  - メモ
  - createdAt
  - updatedAt
- タグモデルを定義する。最低限以下の項目を持つ。
  - 一意な識別子
  - タグ名
  - アイコン識別子
  - sortOrder
  - systemTag フラグ
- 月予算モデルを定義する。最低限以下の項目を持つ。
  - 対象月を表すキー（形式は YYYY-MM とする）
  - 予算金額
- 初期タグを永続化前提で用意する。
  - コンビニ
  - カフェ
  - 外食
  - おやつ
  - 飲み会
  - 娯楽
  - 衝動買い
  - 未分類
- `未分類` をシステム固定タグとして扱う。systemTag = true とし、通常タグとは区別できるようにする。
- 新しい月を初めて扱うときは、直前の月予算を引き継いだ月予算を初期値として導出し、その月の月予算が未保存なら自動で保存する。
- タグ削除時に、そのタグに紐づく既存支出を `未分類` へ再割り当てできるようにする。
- 選択中の月を保存し、アプリ再起動後にも復元できるようにする。
- 支出の作成・更新・削除、タグの作成・更新・削除・並び替え、月予算の作成・更新、選択中月の取得・更新を扱う Repository / Storage 抽象を用意する。
- 後続の F-XX が参照しやすいように、月単位の集計取得を可能にする。
  - 対象月の予算
  - 対象月の支出一覧
  - 対象月の支出合計
  - 対象月の残り金額

---

### 対象外（Out of Scope）
- 画面表示やモーダル表示
- バリデーションメッセージの UI 表示
- タブバーやナビゲーション UI
- グラフや分析機能
- サーバー保存や複数端末同期

---

### 受け入れ条件（Acceptance Criteria）
- [ ] 月予算、タグ、支出、選択中の月、タグ並び順をローカルに保存できる
- [ ] アプリ再起動後も保存したデータを復元できる
- [ ] 支出モデルに tag / amount / date / memo / createdAt / updatedAt が保持される
- [ ] タグモデルに name / icon identifier / sortOrder / systemTag が保持される
- [ ] 月予算は YYYY-MM 形式の月キーで管理される
- [ ] 新しい月を初めて参照したとき、直前月の予算を引き継いだ月予算の初期値を導出できる
- [ ] タグ削除時に、関連支出が `未分類` に再割り当てされる
- [ ] 選択中の月を保存・復元できる
- [ ] 後続機能から利用可能な Repository / Storage 抽象が提供される
- [ ] 対象月の支出合計・残り金額を取得できる

---

### エッジケース（Edge Cases）
- 初回起動時にデータが1件も存在しない場合でも初期タグと空データで動作できること
- 直前月が存在しない状態で新しい月を生成する場合の初期値処理
- `未分類` タグが欠損しないこと
- タグ削除時に大量の支出が存在しても整合性が崩れないこと
- createdAt と updatedAt が更新時に正しく扱われること
- 年をまたぐ月キーでも正しく保存・取得できること

---

### 技術制約（Technical Constraints）
- ローカル保存のみを前提とし、ネットワーク依存を持たない
- MVVM 前提で、View に永続化ロジックを書かない
- 永続化アクセスは Repository / Storage 経由に集約する
- 依存ライブラリの追加は禁止する
- `scripts/` ディレクトリは変更しない
- 日本円・整数金額のみを扱う

---

### テスト要件（Test Requirements）
- 初期データ生成のテスト
- 月予算の作成・取得・更新・初期値導出テスト
- タグの作成・取得・更新・削除・並び替えテスト
- 支出の作成・取得・更新・削除テスト
- タグ削除時の `未分類` 再割り当てテスト
- 月単位の支出合計・残り金額取得テスト
- 選択中月の保存・復元テスト
- アプリ再起動相当の読み込み復元テスト

---

- 現在月の予算が設定済みの場合はホーム画面へ遷移できる前提のルーティング骨格を用意する。
- 以下の基本画面を空状態で用意する。
  - ホーム
  - 支出一覧
  - 分析
  - 設定
- 画面下部にタブバーを表示する。
  - ホーム
  - 支出一覧
  - 分析
  - 設定
- タブ切り替えで各画面を行き来できるようにする。
- 各画面は後続 F-XX の実装前でもクラッシュせず表示できるようにする。
- 各画面には後続機能が未実装でも成立する最低限のプレースホルダー表示を許可する。
- F-04 の月ナビゲーション、F-05 以降の各画面機能、および後続の Premium / 分析機能を載せられる構造にする。

---

### 対象外（Out of Scope）
- 月予算入力そのもの
- 支出登録や支出編集
- タグ管理の中身
- 一覧表示の完成形
- ホーム画面の完成形
- 分析画面の完成形
- Premium 導線の完成形

---

### 受け入れ条件（Acceptance Criteria）
- [ ] アプリ起動後の基本ルートが定義されている
- [ ] 空のホーム・支出一覧・分析・設定画面が表示できる
- [ ] タブバーに「ホーム」「支出一覧」「分析」「設定」が表示される
- [ ] タブ切り替えで各画面を行き来できる
- [ ] 後続機能が未実装でも各画面がクラッシュせず表示できる
- [ ] 現在月の予算が未設定の場合に月予算設定画面へ進める前提の骨格がある

---

### エッジケース（Edge Cases）
- 初回起動判定が未確定でも暫定プレースホルダーで遷移確認できること
- タブ切り替え直後に表示内容が空でも成立すること

## F-03 カラー定義

### 目的（Goal）

Nokori 全体の色表現を一元管理し、無駄遣い抑制アプリとしての落ち着きと視認性を両立する。暗めの青系を基調にしつつ、残額・警告・削除などの重要状態が明確に判別できる配色基盤を定義する。

---

### ユーザーストーリー（User Story）

ユーザーとして、
毎日見ても疲れにくく、残額や警告状態が直感的に分かる、
落ち着いた配色のアプリを使いたい。

---

### 対象範囲（Scope）
- アプリ全体で利用する共通カラー定義を用意する。
- 色は View ごとの直接指定ではなく、共通定義経由で参照する。
- Nokori の基調色は暗めの青系とする。
- 最低限以下の役割色を定義する。
  - Primary
  - Secondary
  - Background
  - Surface
  - TextPrimary
  - TextSecondary
  - Border
  - Warning
  - Danger
  - Success
- 初期カラー値は以下とする。
  - Primary: `#1E3A5F`
  - Secondary: `#3B5B7A`
  - Background: `#F8FAFC`
  - Surface: `#FFFFFF`
  - TextPrimary: `#0F172A`
  - TextSecondary: `#475569`
  - Border: `#CBD5E1`
  - Warning: `#F59E0B`
  - Danger: `#DC2626`
  - Success: `#1D4ED8`
- 各色の役割は以下を前提とする。
  - Primary: 主要見出し、選択中タブ、主要アクション、月表示
  - Secondary: 補助ラベル、非選択アイコン、通常チップ
  - Background: 画面全体背景
  - Surface: カード、行、設定項目背景
  - TextPrimary: 主要テキスト
  - TextSecondary: 補助テキスト
  - Border: 区切り線、枠線
  - Warning: 注意喚起、入力警告、予算接近
  - Danger: 予算オーバー、削除、強い警告
  - Success: 健全な残額表示や補助的な正常状態
- 後続の F-XX はこのカラー定義を前提として画面実装を行う。

---

### 対象外（Out of Scope）
- ダークモード専用パレット
- ユーザーによるテーマ切り替え
- 季節テーマやイベントテーマ
- ブランド刷新を伴う大幅な色変更 UI

---

### 受け入れ条件（Acceptance Criteria）
- [ ] 共通カラー定義を参照できる
- [ ] View 内で直接色をハードコードしない
- [ ] Primary / Secondary / Background / Surface / Text / Border / Warning / Danger / Success が定義されている
- [ ] 初期カラー値が仕様どおりに設定されている
- [ ] 予算オーバー表示に Danger を使える
- [ ] 警告表示に Warning を使える
- [ ] 後続画面がこの定義を参照して実装できる

---

### エッジケース（Edge Cases）
- 警告色と通常色の見分けがつきにくい場合
- 背景色と文字色のコントラスト不足
- 境界線が淡すぎてカード区切りが不明瞭になる場合
- 残額強調時に他要素より視認性が落ちる場合

---

### 技術制約（Technical Constraints）
- 色定義はアプリ内で一元管理する
- View に直接 `Color(...)` を散在させない
- 後続機能が差し替えやすいよう、役割名ベースで色を参照する
- 依存ライブラリの追加は禁止する

---

### テスト要件（Test Requirements）
- 共通カラー定義が取得できること
- 各役割色が仕様どおりの値を持つこと
- Warning / Danger / Success が区別して利用できること
- 主要画面で色定義を参照できること

## F-04 グローバル月ナビゲーション

### 目的（Goal）

アプリ全体で現在参照している月を統一し、ホーム・支出一覧・設定画面・分析画面で同じ月のデータを参照できるようにする。各画面が同じ選択中月に追従することで、Codex が後続画面を順番実装しやすい共通基盤にする。

---

### ユーザーストーリー（User Story）

ユーザーとして、
ホームや支出一覧、設定画面、分析画面の月が常に同期していてほしい。別の月に切り替えたらすべての画面の表示が同じ月に変わり、未来の月には移動できないようにしてほしい。

---

### 対象範囲（Scope）
- アプリ全体で共有される「選択中の月」状態を扱う。
- 月ナビゲーションコンポーネントを作成し、後続の画面から再利用できるようにする。
- コンポーネントは以下を含む。
  - 前月ボタン
  - 次月ボタン
  - 現在の月表示（`YYYY年M月`）
- 前月 / 次月ボタンで月を切り替えられる。
- 左右スワイプでも月を切り替えられる。
- 現在月より未来には移動できない。次月ボタンは無効または非表示にする。
- 月を切り替えると、グローバルな選択中月状態が更新される。
- ホーム、支出一覧、設定画面、分析画面は同じ選択中月状態を参照する前提にする。
- 編集中で未保存の内容があるときに月変更が要求された場合、各画面が破棄確認ダイアログを表示できるようにするためのフックを用意する。破棄確認ダイアログそのものの表示責務は各画面側に持たせる。
- アプリ起動時は F-01 に保存された選択中月、なければ現在月を採用する。

---

### 対象外（Out of Scope）
- 年単位や四半期単位の集計
- 日付範囲の自由指定
- 画面ごとに独立した月状態

---

### 受け入れ条件（Acceptance Criteria）
- [ ] 選択中の月をアプリ全体で共有できる
- [ ] 前月 / 次月ボタンで月を切り替えられる
- [ ] 左右スワイプで月を切り替えられる
- [ ] 現在月より未来には移動できない
- [ ] 月表示が `YYYY年M月` 形式で表示される
- [ ] どこかの画面で変更した月が他画面でも共有される
- [ ] 未保存編集がある状態で月変更を試みた際、各画面が破棄確認ダイアログを表示できる前提の制御ができる
- [ ] アプリ再起動後に最後の選択中月または現在月が初期表示される

---

### エッジケース（Edge Cases）
- 年をまたぐ月変更
- タイムゾーンや端末日付変更時の現在月判定
- 連続スワイプ時の過剰な月更新
- 現在月でさらに次月へ進もうとした場合

---

### 技術制約（Technical Constraints）
- グローバル月状態は再利用可能な状態管理オブジェクトで扱う
- View に月計算ロジックを書かない
- F-01 の保存済み選択中月を利用する
- ジェスチャー競合が起きないようにする

---

### テスト要件（Test Requirements）
- 前月 / 次月ボタンでの月変更テスト
- スワイプでの月変更テスト
- 未来月へ移動できないテスト
- 年またぎの月変更テスト
- グローバル共有状態の同期テスト
- 選択中月の復元テスト

## F-05 月予算の設定・編集

### 目的（Goal）

F-01 の永続化基盤と F-04 の月共有状態を使って、現在月の予算が未設定の場合の予算設定と、設定画面での月別予算編集 UI を実装する。Codex がこの F 単体で「予算に関する画面フロー」を完結して実装できる形にする。

---

### ユーザーストーリー（User Story）

ユーザーとして、
つい使ってしまう支出の残り予算を把握したいため、
現在月の予算が未設定のときに今月の予算を設定し、あとから月ごとの予算を変更したい。

---

### 対象範囲（Scope）
- 現在月の予算が未設定の場合に月予算設定画面を表示する。
- 月予算設定画面ではやさしいトーンの説明文を表示する。
  - 例: 「つい使ってしまう支出を記録して、今月あといくら使えるかを見える化します」
- 月予算設定画面の入力欄には初期値を入れない。
- 月予算設定画面はスキップ不可とする。
- 月予算入力は数値のみ、1円以上999,999円以下の整数を受け付ける。
- 空欄、0円、上限超過は保存不可とし、入力欄からフォーカスが外れた時または保存操作時に、入力欄付近へエラー表示を行う。
- 保存後は今月予算として保存し、ホーム画面へ進める。
- 設定画面から月予算編集画面へ遷移できる。
- 月予算編集画面では F-04 の月ナビゲーションを使って対象月を切り替えられる。
- 月予算編集画面でも未来月には移動できない。
- 月予算を更新して保存できる。
- 新しい予算がその月の支出合計を下回る場合は警告表示するが保存は許可する。
- 編集中に閉じる、または月を切り替えるとき未保存の内容があれば破棄確認ダイアログを表示する。
- 保存成功後はホーム・支出一覧・分析画面の表示に反映される前提とする。
- 保存失敗時は画面上部または下部に共通エラーメッセージを表示する。

---

### 対象外（Out of Scope）
- タグ別予算
- 複数通貨対応
- 収入や固定費を含む家計簿機能
- クラウド同期

---

### 受け入れ条件（Acceptance Criteria）
- [ ] 現在月の予算が未設定の場合に月予算設定画面が表示される
- [ ] 説明文がやさしいトーンで表示される
- [ ] 入力欄に初期値が入っていない
- [ ] 数値のみ入力できる
- [ ] 1〜999,999円以外は保存できず、エラー表示される
- [ ] 月予算設定画面はスキップできない
- [ ] 保存すると今月予算として保存され、次画面へ進める
- [ ] 設定画面から月別予算を編集できる
- [ ] 月切り替えで未来月へ移動できない
- [ ] 支出合計未満の予算を保存しようとすると警告が表示されるが保存は可能
- [ ] 未保存の状態で閉じる・月を変える場合は破棄確認が表示される

---

### エッジケース（Edge Cases）
- 支出合計が既に存在する月を下回る予算へ変更するケース
- 先頭ゼロや空白入力
- 1,000,000円以上の入力
- 月切り替え中の未保存状態
- 現在月の予算未設定判定に必要な永続化読み込みが失敗するケース

---

### 技術制約（Technical Constraints）

## F-11 Premium 機能制限と訴求導線

### 目的（Goal）

Nokori の無料利用と Premium 利用の機能境界を定義し、継続利用に価値のある機能をサブスクリプション対象として扱う。無料ユーザーにも通常利用は維持しつつ、全期間の履歴閲覧、残額ペース予測、月比較を Premium 機能として提供し、自然な訴求導線を実装できる状態にする。

---

### ユーザーストーリー（User Story）

ユーザーとして、
無料でも日常的な記録は続けられつつ、
より深い振り返りや分析をしたくなったら Premium に進みたい。

---

### 対象範囲（Scope）
- 無料ユーザーと Premium ユーザーの機能差分を定義する。
- 無料ユーザーの履歴閲覧範囲は当月を含めて3ヶ月までとする。
- Premium ユーザーは全期間の履歴を閲覧できる。
- 閲覧制限は過去方向にのみ適用し、未来月は従来どおり全ユーザー共通で閲覧不可とする。
- 無料ユーザーが制限対象の月へ移動しようとした場合、月移動は行わず、その場で Premium 訴求モーダルを表示する。
- 制限対象はホーム画面と支出一覧画面の両方とする。
- Premium 機能は以下の3つとする。
  - 全期間の履歴閲覧
  - 残額ペース予測
  - 月比較
- 設定画面に「Premium」導線を1行の設定項目として表示する。右側の状態表示は「未加入 / 利用中」とする。
- 設定画面の「Premium」行は、未加入 / 利用中どちらでも Premium 詳細画面へ遷移する。
- 分析タブに関する無料 / Premium の分岐表示ルールは後続の F-13 で定義し、本セクションでは課金対象機能と訴求導線のルールのみを扱う。
- Premium 訴求モーダルのタイトルは「Premium」とする。
- Premium 訴求モーダルの本文は「Premiumで全期間の履歴閲覧・残額ペース予測・月比較が使えます」とする。
- Premium 訴求モーダルには以下をこの順で表示する。
  - 全期間の履歴閲覧
  - 残額ペース予測
  - 月比較
- Premium 訴求モーダルのボタンは「Premiumを見る」「閉じる」とする。
- 「Premiumを見る」を押した時は Premium 詳細画面へ遷移する。

---

### 対象外（Out of Scope）
- Premium 詳細画面のレイアウトや購読プラン表示
- 購入 / 復元 / 状態同期の実装詳細
- 分析タブ内の無料 / Premium 分岐表示
- 買い切りプラン
- Web 決済や App Store 以外の決済導線

---

### 受け入れ条件（Acceptance Criteria）
- [ ] 無料ユーザーの履歴閲覧範囲が当月を含めて3ヶ月までに制限される
- [ ] Premium ユーザーは全期間の履歴を閲覧できる
- [ ] 無料ユーザーが制限月へ移動しようとした時に Premium 訴求モーダルが表示される
- [ ] 制限対象がホーム画面と支出一覧画面の両方に適用される
- [ ] 設定画面に「Premium」導線が表示され、状態表示が「未加入 / 利用中」となる
- [ ] 分析タブに関する無料 / Premium の分岐表示ルールが F-13 と整合している
- [ ] Premium 訴求モーダルに3機能が指定順で表示される
- [ ] 「Premiumを見る」で Premium 詳細画面へ遷移できる

---

### エッジケース（Edge Cases）
- 無料ユーザーが4ヶ月以上前へ移動しようとしてモーダルを閉じた場合、元の表示月を維持すること
- 購読状態反映直後にホームや支出一覧の表示可能範囲が即時変化すること

---

### 技術制約（Technical Constraints）
- 無料 / Premium の機能境界は役割ベースで判定できるようにする
- 履歴制限到達時の訴求導線は共通モーダルで扱う
- 訴求文言や表示順は本セクションの仕様に従って一元管理する
- 依存ライブラリ追加は禁止する

---

### テスト要件（Test Requirements）
- 無料ユーザーの3ヶ月制限テスト
- Premium ユーザーの全期間閲覧テスト
- 制限到達時の Premium 訴求モーダル表示テスト
- 設定画面の Premium 導線・状態表示テスト
- 訴求モーダルから Premium 詳細画面への遷移テスト

## F-12 Premium 詳細画面と購読状態管理

### 目的（Goal）

Premium 詳細画面、購読プラン表示、購入 / 復元 / 管理導線、購読状態同期を定義し、ユーザーが Premium の内容を理解し、App Store 経由で購読状態を安全に管理できるようにする。

---

### ユーザーストーリー（User Story）

ユーザーとして、
Premium の内容と現在の購読状態を確認し、
登録・復元・管理を分かりやすく行いたい。

### 対象範囲（Scope）
- Premium 詳細画面を実装する。画面タイトルは「Premium」とする。
- Premium 詳細画面には3機能の説明を表示する。
- 購読 UI には StoreKit の SubscriptionStoreView を使用する。
- 3機能の表示順は F-11 の Premium 訴求モーダルと同じにする。
- 3機能の説明文は以下とする。
  - 全期間の履歴閲覧: 「過去3ヶ月を超える履歴も確認できます」
  - 残額ペース予測: 「月末予測残額と1日あたり使える目安を確認できます」
  - 月比較: 「選択中の月と前月の支出合計・残額・タグ別増減を比較できます」
- SubscriptionStoreView 上で月額プランと年額プランを表示する。
- 年額プランには「おすすめ」バッジを表示する。
- 価格表示は App Store から取得した実際の価格をそのまま表示する。
- 購読導線の中心は SubscriptionStoreView に委ねる。個別の購読ボタン文言は SubscriptionStoreView の標準表示に従う。
- Premium 詳細画面では現在の加入状態と加入中プラン名を表示する。表示文言の例は「利用中（月額）」「利用中（年額）」とする。
- 加入中プランは「利用中」と表示し、押せないようにする。
- 加入中はもう一方のプランも表示するが、押せない状態とする。
- Premium 詳細画面に「プラン変更は App Store のサブスクリプション管理から行えます」という補足文言を表示する。
- Premium 詳細画面と設定画面の両方に「サブスクリプションを管理」導線を置く。
- 「サブスクリプションを管理」を押した時は App Store のサブスクリプション管理画面を開く。
- Premium 詳細画面と設定画面の両方に「購入を復元」導線を置く。
- Premium 機能の有効判定は App Store の現在の購読状態に基づいて行う。
- Premium 購入完了後は Premium 機能を即時反映する。
- 購入復元成功後も Premium 機能を即時反映する。
- Premium 解約後は、有効期限が切れた時点で無料制限に戻る。
- Premium 状態取得に失敗した時は安全側で無料ユーザーとして扱う。
- 購入失敗時や購入復元失敗時は、その場にエラーメッセージを表示する。

---

### 対象外（Out of Scope）
- 無料 / Premium の履歴制限ルール
- 制限到達時の訴求モーダル仕様
- 分析タブ内のロック画面表示
- ファミリー共有などの高度な課金オプション
- クーポンや割引機能

### 受け入れ条件（Acceptance Criteria）
- [ ] Premium 詳細画面に3機能説明と SubscriptionStoreView による購読 UI が表示される
- [ ] 年額プランに「おすすめ」バッジが表示される
- [ ] App Store の実価格が表示される
- [ ] 加入状態と加入中プラン名が表示される
- [ ] 加入中プランともう一方のプランが押せない状態で表示される
- [ ] 「サブスクリプションを管理」で App Store のサブスクリプション管理画面を開ける
- [ ] 「購入を復元」が設定画面と Premium 詳細画面の両方に表示される
- [ ] 購入・復元成功後に Premium 機能が即時反映される
- [ ] 解約後は有効期限切れ時に無料制限へ戻る
- [ ] Premium 状態取得失敗時に安全側で無料ユーザーとして扱われる
- [ ] 購入失敗 / 購入復元失敗時にその場でエラーメッセージが表示される

### エッジケース（Edge Cases）
- App Store から一時的に価格取得や状態取得が失敗すること
- Premium 解約済みだが有効期限内の期間は Premium として扱われること
- 購入状態反映直後にホームや支出一覧の表示可能範囲が即時変化すること
- 加入中プランと未加入プランが同時に表示されること

### 技術制約（Technical Constraints）
- App Store の購読状態を参照して Premium 有効判定を行う
- 購読 UI は可能な限り StoreKit 標準の SubscriptionStoreView を利用する
- 価格表示は App Store 取得値を使用する
- 購読 / 復元 / 状態同期の結果は UI に即時反映する
- 課金状態取得失敗時は安全側で無料ユーザー扱いにする
- 依存ライブラリ追加は禁止する

## F-13 分析画面

### 目的（Goal）

Premium ユーザー向けの分析画面を提供し、残額ペース予測と月比較を1画面で確認できるようにする。分析タブ内での無料 / Premium の表示分岐と、Premium ロック画面から Premium 詳細画面へ遷移できる導線を定義する。

---

### ユーザーストーリー（User Story）

ユーザーとして、
今月の支出ペースや前月との違いをまとめて確認して、
無駄遣いを振り返りたい。

---

### 対象範囲（Scope）
- 分析画面を実装する。画面タイトルは「分析」とする。
- タブバーに「分析」タブを追加する。並び順は「ホーム → 支出一覧 → 分析 → 設定」とする。
- 分析タブは全ユーザーに表示する。
- 分析タブ内での無料 / Premium の分岐表示を本セクションで定義する。
- Premium ユーザーが分析タブを押した場合は、分析コンテンツを表示する。
- 無料ユーザーが分析タブを押した場合も分析タブへは遷移し、分析タブ内に Premium ロック画面を表示する。Premium ロック画面内には Premium 詳細画面へ遷移できる導線を表示する。
- Premium ロック画面には閉じる操作を置かず、分析タブ内のロック状態として表示する。別タブへの移動は通常のタブ切り替えで行う。
- 分析画面上部には F-04 の月ナビゲーションを表示する。
- 分析画面はグローバル選択月に同期する。
- 月を切り替えると、残額ペース予測と月比較の両方をその月基準で更新する。
- 分析画面の構成順は「残額ペース予測 → 月比較」とする。
- 残額ペース予測では以下を表示する。
  - 月末予測残額
  - 1日あたり使える目安
  - 今のペース判定
- 今のペース判定は3段階とする。
  - 予算内
  - やや使いすぎ
  - オーバー見込み
- 月末予測残額は「当月の累計支出 ÷ 経過日数 × 月の日数」で月末支出を予測し、予算との差を表示する。
- 1日あたり使える目安は「現在の残額 ÷ 残り日数」で計算する。
- 経過日数・残り日数は今日を含めて数える。
- 残額ペース予測は月初1日目から表示する。1日目はその日の支出のみをもとに計算する。
- 月比較は分析画面の中だけに表示する。
- 月比較は選択中の月と前月を比較する。
- 月比較では以下を表示する。
  - 月の支出合計
  - 残額
  - タグ別増減
- タグ別増減は、増減の大きい順に上位3件を表示する。

---

### 対象外（Out of Scope）
- Premium ロック画面の詳細なマーケティング文言の最適化
- 先月以外の任意月比較
- 3ヶ月平均や長期トレンド分析
- グラフの高度なカスタマイズ

---

### 受け入れ条件（Acceptance Criteria）
- [ ] タブバーに「分析」が表示される
- [ ] タブの並び順が「ホーム → 支出一覧 → 分析 → 設定」となる
- [ ] Premium ユーザーの分析タブでは分析コンテンツが表示される
- [ ] 無料ユーザーが分析タブへ遷移すると、分析タブ内に Premium ロック画面が表示される
- [ ] 無料ユーザーの分析タブでは、Premium ロック画面がロック状態として表示される
- [ ] 分析画面に月ナビゲーションが表示される
- [ ] 分析画面がグローバル選択月に同期する
- [ ] 分析画面の構成順が「残額ペース予測 → 月比較」となる
- [ ] 残額ペース予測に月末予測残額・1日あたり使える目安・3段階のペース判定が表示される
- [ ] 経過日数・残り日数が今日を含めて計算される
- [ ] 月初1日目から残額ペース予測が表示される
- [ ] 月比較に支出合計・残額・タグ別増減上位3件が表示される
